## 一、调研背景与目标

### 1.1 调研背景

**1. 考研市场规模仍处高位，需求结构正在变化**

教育部数据显示，2025 年全国硕士研究生报考人数为 388 万，较 2024 年的 438 万下降约 11.4%。报考总量虽有回落，但报录比仍处于较高水平，热门专业竞争激烈。同时，2024 年二战及以上往届考生占比达 51.82%，首次超过应届生，复读群体对"高效提分"与"精准复习"的需求显著上升（中国教育在线，2025）。综合来看，考研市场已从"规模扩张"阶段进入"效率竞争"阶段——用户对学习效率与方法论的需求高于以往。

**2. 传统考研学习模式存在效率瓶颈**

公开讨论与教育媒体反馈中，常见的痛点是"看书时觉得懂了，做题或答主观题时却答不到点上"。这种"理解幻觉"在以输入式学习为主的备考模式中较为普遍，输出训练相对不足是导致学习效果不理想的常见原因之一。

**3. 大语言模型与 AI Agent 技术为个性化学习提供新可能**

近年大语言模型在知识讲解、答疑、批改等场景的应用日趋成熟。AI Agent 技术的发展（LLM + 工具调用 + 记忆 + 规划），使得"智能陪练"与"个性化反馈"在工程上具备可行性，为传统费曼学习法的应用难点提供了潜在解决路径。

**4. 费曼学习法的认知与应用缺口**

费曼学习法已为相当一部分考研学生所了解，但据公开社区讨论反映，能长期坚持使用的用户比例较低。主要障碍集中在两点：缺少合适的陪练对象、缺少对"自己讲得对不对"的判断依据。

### 2.1 基础属性

| 维度 | 主要细分 |
|------|---------|
| 年龄 | 21–23 岁（应届毕业生）、24–26 岁（二战 / 在职备考）、27 岁及以上（大龄考生） |
| 学科 | 公共课（政治 / 英语 / 数学）、专业课（统考 / 自命题） |

### 2.2 用户关注优先级

考研用户的关注点大致可分为三层：

- **一级关注**：知识点理解深度、主观题得分能力、复习精准度、单位时间提分效率
- **二级关注**：错题复盘效率、背诵记忆效果、学习计划合理性、时间管理
- **三级关注**：学习动力、信息获取、同伴交流、学习环境

费曼学习法的发力点主要在一级关注层。

### 2.3 典型用户分群（按核心学习痛点）

| 用户类型 | 特征 | 典型行为 | 核心需求 |
|---------|------|---------|---------|
| 概念理解困难型 | 跨考 / 基础薄弱，对抽象原理似懂非懂 | 反复看网课仍不能应用 | 通俗拆解、建立知识体系 |
| 输出能力薄弱型 | 背得熟，但主观题写不到点 | 背诵流利，答题逻辑混乱 | 训练语言组织与表达 |
| 知识碎片化型 | 单点记住但无法贯通 | 选择题强，综合题弱 | 重建知识关联 |
| 盲目刷题型 | 量大但不总结 | 错题本只抄题不分析 | 精准定位知识漏洞 |
| 自律性差型 | 计划难执行 | 学习时长长但效率低 | 监督与反馈 |

## 三、考研用户核心痛点（按费曼适配性分类）

为对照费曼学习法的解决能力，本节将考研痛点按学习过程的三阶段重新组织。

### 3.1 输入阶段：理解幻觉与碎片化

- **理解幻觉**：看书时觉得逻辑通顺，合上书却无法用自己的话讲清楚——输入式学习最典型的副作用，用户获得的是"熟悉感"而非真正理解
- **概念模糊**：对专业术语和抽象原理只能死记结论，无法解释本质（典型如数学定理推导、经济学模型、法学条文）
- **知识碎片化**：知识点孤立存在，无法建立前后逻辑联系
- **信息过载**：辅导资料众多，难以判断哪些是核心内容

### 3.2 内化阶段：缺乏有效反馈

- **缺乏反馈**：自己讲给自己听，无法识别理解漏洞与逻辑跳跃
- **反馈不专业**：找同学陪练，但同伴水平有限，难以精准指出知识错误
- **迁移能力差**：知识点听懂了，无法应用到具体题目中
- **复习无重点**：不清楚自己哪些知识点掌握不牢，只能全面复习，时间浪费

### 3.3 输出阶段：主观题与表达瓶颈

- **答题逻辑混乱**：会背知识点，但不会组织语言，答题抓不住得分点（考研政治主观题约占 50%，专业课主观题占比 60–100%）
- **表达不精准**：写了很多但击不中评分标准的核心
- **记忆不牢固**：被动背诵容易遗忘
- **答题速度慢**：平时缺乏输出训练，考试时来不及

### 3.4 传统费曼学习法的应用局限

公开讨论中常见的反馈是，费曼学习法本身被认为有效，但难以在考研场景中持续执行，原因集中在：

1. **缺少陪练对象**：很难找到愿意耐心听讲解并持续追问的人
2. **时间成本高**：每次讲解都需要占用他人时间，无法随时进行
3. **缺乏量化评估**：不知道自己讲得到底好不好，改进方向不明确
4. **无法形成闭环**：暴露的问题没有自动进入复习计划，再次出现

这四条限制，恰恰是 AI Agent 可以介入的切入点。

## 四、费曼学习法在考研场景的适配性

### 4.1 费曼学习法核心原理

费曼学习法由物理学家理查德·费曼提出，核心逻辑是**输出倒逼输入**：通过"教给别人"暴露理解漏洞。经典四步：

1. **选择概念**：选定一个想掌握的知识点
2. **教给小白**：假设听众是一个完全不懂的人，用最简单的语言讲清楚
3. **发现漏洞**：在卡壳、含糊、绕不开的地方，就是知识漏洞
4. **回顾简化**：回到原始材料，补齐漏洞，再用更简单的语言重讲

### 4.2 费曼学习法解决考研痛点的核心价值

| 考研痛点 | 费曼学习法的解决方式 | 核心价值 |
|---------|------------------|---------|
| 理解幻觉 | 要求用自己的话讲解，快速暴露"似懂非懂" | 显著提升理解深度 |
| 概念模糊 | 用通俗语言解释专业术语 | 避免死记硬背 |
| 知识碎片化 | 通过讲解重构知识体系 | 实现融会贯通 |
| 输出能力弱 | 训练语言组织与逻辑表达 | 提升主观题得分能力 |
| 记忆不牢 | 主动输出比被动记忆留存率更高 | 减少遗忘 |
| 复习盲目 | 通过追问精准定位知识盲区 | 提升复习效率 |

### 4.3 适用场景与边界

**最适用场景**：

- 概念理解：数学定理、专业课抽象概念、政治原理
- 主观题训练：论述、简答、案例分析
- 错题复盘：分析错误原因、理解解题思路
- 知识体系构建：梳理章节逻辑、建立知识框架

**较适用场景**：

- 背诵记忆：通过讲解加深对背诵内容的理解
- 真题解析：讲解解题思路与得分点
- 专题总结：归纳某一专题的知识与方法

**不适用场景**（项目应明确避开）：

- 纯计算训练（数学计算题、英语语法题）
- 纯记忆类（英语单词、历史年代、政治时事）
- 技能类训练（英语口语、听力）

## 五、AI Agent 赋能费曼学习法的应用方向

### 5.1 AI Agent 核心能力

依据 AI Agent 的通用定义：

> **AI Agent = LLM（大脑）+ Planning（规划）+ Tool Use（工具调用）+ Memory（记忆）**

四种能力对费曼学习场景的支撑：

| 能力 | 在费曼场景中的作用 |
|------|----------------|
| LLM | 理解用户讲解、识别问题、生成自然语言追问与反馈 |
| Planning | 根据用户水平制定个性化训练计划 |
| Tool Use | 调用考研知识库、真题、错题本等外部资源 |
| Memory | 记录讲解历史、知识漏洞、复习进度 |

### 5.2 候选赋能方向（不作 MVP 范围决策）

以下五个方向为本周调研归纳出的潜在功能候选。

**A. 智能费曼陪练**

- AI 扮演"零基础听众"，针对用户讲解中的术语、跳跃、含糊处连续追问
- 输入方式（语音 / 文字 / 其他）的具体支持范围由后续工程评估决定
- 不同学科可能需要差异化的追问策略

示例：用户讲解"边际效用递减"时，AI 可能追问"什么是边际效用？能否举生活中的例子？为什么会递减？有没有例外？"

**B. 自动费曼诊断报告**

- 评估维度参考产品愿景文档：**理解深度、表达完整性、逻辑连贯性、结构化能力**
- 具体指出"哪些概念没讲透、哪些地方逻辑跳跃、哪些前提被省略"，避免笼统打分

**C. 个性化复习闭环**

- 讲解中暴露的知识漏洞自动进入复习队列
- 结合艾宾浩斯遗忘曲线或类似机制安排复习
- 定期生成薄弱点分析

**D. 主观题费曼批改**

- 将主观题答案视为一种特殊形式的"讲解"
- 评估得分点覆盖度、逻辑清晰度、表达准确性

**E. 跨考零基础入门支持**

- 用通俗语言拆解专业概念
- 提供生活化类比与例子，降低跨考门槛

### 6.4 按费曼流程的分阶段分析

#### Step 1：学习知识（资料导入与拆解）

样本概念：牛顿第二定律。

- **Kimi 样本**：上传物理课本 PDF → 输出：定义 + 适用条件 + 易错误区 + 生活举例
- **豆包样本**：上传讲义 → 指令 → 拆分重点、高频易错点，提前标注易混淆内容
- **NotebookLM 样本**：上传讲义 → 自动生成章节思维导图，梳理逻辑

#### Step 2：模拟讲课

**Gemini 使用样例 Prompt**：

```text
我来讲牛顿第二定律。请你分析我的知识掌握程度，识别我的知识盲区，
对盲区进行深度解析，然后生成相应的 quiz，帮助我掌握这个知识点。
我开始讲课：F=ma，力等于质量乘速度。
```

**AI 反馈样例**：

> 在您的第一句话中："**F=ma，力等于质量乘速度。**"，已经暴露了一个非常关键的**概念性盲区**。
>
> **正确的部分**：您准确地记住了牛顿第二定律的公式 F=ma，并且知道 F 代表力，m 代表质量。
>
> **知识盲区**：您将公式中的 a 误理解为了"**速度**"（Velocity）。
>
> **盲区本质**：在物理学中，a 代表的是**加速度**（Acceleration），而非速度。这两个概念有着本质的区别，混淆它们会导致无法正确理解力是如何改变物体运动状态的。

**豆包使用样例**：

- 用户（语音口述）：牛顿第二定律 F=ma，力越大速度越大。
- AI：我听不懂啦，速度和加速度不一样，你用跑步举个例子重新讲讲？

**扣子（Coze）**：用户自建「小学生听课智能体」，上传物理知识库后试讲，AI 自动对照文档纠错。

#### Step 3：漏洞排查

用户主动讲述知识点，AI 实时监听、识别概念错误、逻辑漏洞、知识点缺失，定位理解误区。

- **Gemini**：可识别用户口述中的概念错误、逻辑偏差，实时指出问题并解析盲区
- **豆包**：支持实时监听用户语音 / 文字讲解，标记漏洞，归类漏洞类型
- **ChatPDF / NotebookLM**：无口述监听排查能力，仅能依托文档原文被动答疑，无法针对用户的个性化讲述错误查漏
- **扣子 / Dify**：需手动搭建智能体、配置漏洞检测规则后，才可对照知识库排查讲述漏洞，无原生能力
- **Quizlet / Anki**：完全不支持口述漏洞排查，仅聚焦记忆背诵，无法识别理解层面误区

#### Step 4：辅助简化表达

样本概念：热力学第一定律 ΔU = Q + W

- **Gemini 输出**：概念拆解（如 *Q（热量）：相当于外界"给你打钱"或者你"花钱"*），极简一句话总结
- **豆包输出**：结合此前讲错的做功知识点，改成生活化短句，修正逻辑漏洞
- **ChatPDF**：照搬课本定义，简化程度低

#### Step 5：传授验证

**目标**：通过验证机制确认掌握程度。

**现有产品表现**：几乎缺失原生费曼式"再教别人验证"机制。**其他验证掌握程度的方式包括**使用知识卡片复习、出题自测：

- **Quizlet / Anki**：知识卡片自测
- **Gemini**：生成 quiz 题目测试掌握程度，总结正确率和各方面掌握程度、生成知识卡片帮助记忆
- **ChatPDF**：生成知识卡片自测

### 6.5 各产品类型总结

1. **通用大模型（Kimi / 通义千问 / 豆包等）**：可以 AI 输出知识，可以接收用户输出讲课，通过提示词可以支持费曼"以教代学"核心。**缺少原生 SRS 间隔复习**，错题不能自动按遗忘曲线推送复习
2. **RAG 工具（ChatPDF / NotebookLM）**：擅长读文档，**不能练口述**
3. **扣子 / Dify**：功能上限高，但**需要用户自行配置智能体**
4. **卡片工具（Anki / Quizlet）**：聚焦背诵，**无法理解式学习**，解决不了知识理解漏洞

### 6.6 现有产品的核心不足与本项目机会点

通过上述分析可以看到，**现有产品在费曼 5 步流程上存在明显空白**：

- **Step 5（传授验证）几乎全部缺失** —— 所有产品在"再次教别人来验证理解"这一环节都没有原生支持
- **Step 2/3/4（讲解 / 查漏 / 简化）需要依赖提示词手工实现** —— 通用大模型虽能通过 Prompt 实现，但缺乏稳定流程与产品化封装
- **没有产品打通完整闭环** —— 讲解、查漏、复习三件事分别由不同类型产品承担，没有一个产品能跑通完整费曼链路

### 6.7 基于竞品分析的初步设想

参考传统费曼流程"学习 → 讲解 → 回顾 → 简化 → 传授"，结合竞品空白，可考虑的 AI 费曼学习闭环方向为：

```text
导入教材（多格式 / 中英）
   ↓
AI 拆解知识 + 标注易错点
   ↓
用户文字 / 语音口述讲课（AI 扮演零基础学生）
   ↓
AI 实时定位知识漏洞（概念错误 / 逻辑缺失）
   ↓
针对性追问 + 结合原文纠错
   ↓
多维度评分（完整度、准确度、通俗度）
   ↓
AI 辅助简化重构话术
   ↓
个性化出题验证掌握程度
   ↓
错题自动生成专属记忆卡片
   ↓
SRS 间隔周期性复习
   ↓
周期性复盘，优化后续学习重点
```



## 七、算法与技术调研

### 7.1 AI Agent 技术综述

**7.1.1 Agent 的定义与四大组件**

AI Agent 是以大语言模型（LLM）为核心、能够自主感知-决策-行动的智能体系统。当前业界共识的 Agent 架构由四部分组成：

| 组件 | 作用 | 在费曼场景中的对应 |
|------|------|------------------|
| LLM（大脑） | 自然语言理解与生成 | 理解用户讲解、生成追问与反馈 |
| Planning（规划） | 任务拆解与执行步骤编排 | 决定"何时追问、何时评估、何时进入复习" |
| Tool Use（工具调用） | 调用外部 API / 函数 | 检索知识库、写入复习队列、调用打分函数 |
| Memory（记忆） | 短期对话上下文 + 长期用户档案 | 记录讲解历史、知识漏洞、复习进度 |

**7.1.2 主流 Agent 编排范式**

| 范式 | 核心思想 | 典型场景 | 是否适合本项目 |
|------|---------|---------|-------------|
| **ReAct**（Reasoning + Acting） | "思考-行动-观察"循环，由 LLM 动态决策下一步 | 开放式问答、不确定流程 | 部分适合，灵活但难控 |
| **Plan-and-Execute** | 先由 Planner 出完整计划，再由 Executor 顺序执行 | 流程固定的多步任务 | 适合，但费曼流程需要中途回退 |
| **状态机编排（LangGraph 风格）** | 把业务流程建模为节点 + 状态转移 | 流程明确、状态有限的任务 | **最适合费曼模式**（讲解 → 追问 → 评估 → 复习是天然的状态机） |
| **多 Agent 协作** | 多个 Agent 各司其职，由 Orchestrator 协调 | 复杂分工的任务 | V2 考虑（如"追问 Agent"+"评估 Agent"分离） |

**7.1.3 主流 Agent 框架对比**

| 框架 | 出品方 | 优点 | 缺点 | 适合本项目 |
|------|------|------|------|----------|
| **LangChain** | LangChain Inc. | 生态最完整、文档与教程多、中文资料丰富 | 抽象层次较多，被诟病过度封装；版本更新快、有兼容性问题 | ✅ 入门友好 |
| **LangGraph** | LangChain Inc. | 显式状态机编排，可控性强；与 LangChain 生态打通 | 学习曲线略陡，文档相对新 | ✅✅ **首选**，与费曼流程契合 |
| **LlamaIndex** | LlamaIndex Inc. | RAG 能力强、文档索引优秀 | Agent 编排能力较弱 | 部分使用（仅用其 RAG 模块） |
| **AutoGen** | Microsoft | 多 Agent 对话协作能力强 | 偏研究向，工程化程度低 | 不选（过于复杂） |
| **扣子 / Dify** | 字节 / Dify | 低代码可视化搭建 | 难定制深度逻辑，黑盒化严重 | 不选（不利于体现工程能力，简历加分有限） |

**7.1.4 本项目建议**

- **主框架：LangGraph**，理由：
  1. 费曼模式"讲解 → 追问 → 评估 → 进入复习"是天然的状态机，LangGraph 的显式节点+边设计能让流程清晰可控
  2. 与 LangChain 生态打通，可直接使用其 RAG、Tool、Memory 组件
  3. 业界对状态机式 Agent 编排关注度上升，是大厂 JD 高频提及的技术方向
- **备选：LangChain**
- **不选**：扣子 / Dify 等低代码平台，不利于深度定制与工程能力展示

### 7.2 Prompt Engineering 调研

**7.2.1 基础范式**

| 范式 | 说明 | 在本项目中的用途 |
|------|------|---------------|
| Zero-shot | 仅给指令，不给例子 | 简单指令场景 |
| Few-shot | 给若干示例引导 LLM 学习格式 | 追问示例、评估输出格式 |
| CoT（思维链） | 让 LLM 显式写出推理过程 | 评估打分时的依据分析 |
| Self-Consistency | 多次采样取多数 | 关键评估节点的鲁棒性提升 |
| ReAct Prompt | "思考-行动-观察"格式 | Agent 编排的底层 prompt 模板 |

**7.2.2 System Prompt 设计原则**

针对费曼模式"不直接给答案、只追问的听众"角色，System Prompt 需明确以下几点：

1. **角色定义**：扮演零基础听众，对用户讲解的内容毫无背景知识
2. **行为约束**：禁止直接给出答案；禁止评价"对/错"，只能以追问推动用户自我修正
3. **追问策略**：基于产品愿景定义的 5 类追问动作（解释为什么、换简单说法、举例、对比相近概念、追问被忽略的条件）
4. **输出格式**：约束为结构化字段（追问内容 + 触发的诊断类型）

**7.2.3 结构化输出**

为支持自动评估与复习闭环写入，关键节点需要 LLM 返回结构化 JSON 而非自由文本。三种实现方式对比：

| 方式 | 实现成本 | 可靠性 | 备注 |
|------|--------|--------|------|
| Prompt 强约束 + JSON 模式 | 低 | 中（偶尔格式错误） | DeepSeek、通义均支持 `response_format: json_object` |
| Function Calling | 中 | 高 | 主流 API 均支持，**推荐** |
| Pydantic + LangChain Output Parser | 中 | 高 | 工程友好，便于类型检查 |

**7.2.4 与项目核心要素的映射**

| 项目要素 | Prompt 设计落点 |
|---------|---------------|
| 5 类诊断（复述/跳跃/模糊/省略/空泛） | 在追问 Agent 的 System Prompt 中定义检测规则，输出 `detected_issue` 字段 |
| 5 类追问动作 | 在追问 Agent 中定义动作集合，由 LLM 选择最匹配的一种 |
| 4 维评估（理解深度/表达完整/逻辑连贯/结构化能力） | 独立的评估 Agent，输出结构化 JSON：`{"理解深度": int, "理由": str, ...}` |
| 复习闭环 | 评估完成后，由 Tool Use 调用写入数据库 |

**7.2.5 本项目建议**

- 系统采用**多 Prompt 分工**：追问 Prompt 与评估 Prompt 各自独立、解耦
- 评估环节使用 **Function Calling + JSON Schema** 保证结构化输出可靠
- 追问环节支持 **Few-shot 示例库**，可按学科切换示例（政治 / 专业课 / 数学定理等）

### 7.3 RAG 知识库检索增强生成

**7.3.1 RAG 全流程**

```
文档输入 → [1] 解析 → [2] 切片 → [3] Embedding → [4] 入向量库
                                                        ↓
用户提问 → [5] Query Embedding → [6] 向量检索 → [7] Rerank → [8] 拼接上下文 → [9] LLM 生成
```

**7.3.2 关键环节技术选型**

**(1) 切片策略**

| 策略 | 说明 | 优点 | 适合场景 |
|------|------|------|---------|
| 固定长度切片 | 按字符 / token 数切 | 简单 | 通用文档 |
| 递归切片（Recursive） | 按段落-句子-字符层级切 | 保留语义边界 | **推荐**，LangChain 默认 |
| 语义切片 | 用 embedding 判断语义断点 | 切片质量最高 | 文档质量要求极高时 |
| 按结构切片 | 按 Markdown / HTML 标题切 | 适合教材类 | **考研教材首选** |

考研教材有清晰的章节-小节-知识点结构，建议**按结构切片为主，递归切片为辅**。

**(2) Embedding 模型**

| 模型 | 出品方 | 语言 | 部署方式 | 价格 | 适合本项目 |
|------|------|------|--------|------|----------|
| **BGE-large-zh-v1.5** | 北京智源 | 中文 SOTA | 本地 / API | 免费（开源） | ✅✅ **首选** |
| m3e-base | MokaAI | 中文 | 本地 | 免费 | 备选 |
| text-embedding-3 | OpenAI | 英文 SOTA | API | 收费、国内访问困难 | ❌ |
| 通义 / DeepSeek Embedding | 阿里 / 深度求索 | 中文 | API | 低 | 可考虑（省去本地部署） |

**(3) 向量数据库**

| 向量库 | 优点 | 缺点 | 适合阶段 |
|--------|------|------|---------|
| **Chroma** | 轻量、Python 原生、API 简单、可本地或 client-server | 大规模性能不及 Milvus | **当前 首选** |
| FAISS | 检索性能强、Meta 出品 | API 偏底层、不带持久化方案 | 性能优化阶段 |
| Milvus | 工业级、分布式、生态完整 | 重，部署运维复杂 | 上线后扩展 |
| Pinecone / Weaviate | 云服务、免运维 | 收费、数据出境合规问题 | 不选 |

**(4) Rerank（可选但建议）**

引入 **BGE-reranker** 在向量检索后做二次排序，能显著提升 top-k 检索的相关性，对答主观题这种"需要召回多个相关知识点并精准排序"的场景效果尤其明显。

**7.3.3 考研知识库的内容构建策略（与 MVP 范围相关）**

知识库的具体内容范围与 MVP 用户 / 学科聚焦决策强相关，**最终范围由后续集体讨论共同确定**。本节仅讨论**通用构建原则**与**不同范围下的实施差异**：

- **通用构建原则**：
  - **数据来源候选**：考研大纲解析、公开讲义、知识点笔记
  - **切片粒度**：以"知识点"为基本单位（约 200–500 字），保留章节层级元数据
  - **元数据设计**：每个切片附带 `{科目, 章节, 知识点名, 难度, 题型适配}` 等结构化字段，支持按维度过滤检索
- **不同范围下的实施差异（供后续讨论）**：
  - **若聚焦单一学科**（如政治或某门专业课）：数据规模小、切片质量易把控
  - **若覆盖多学科公共课**（政治 + 英语 + 数学）：需考虑跨学科 metadata 体系、Embedding 对不同领域文本的适配
  - **若覆盖自命题专业课**：资料获取难度较大，需要更长的数据准备周期

**7.3.4 本项目技术栈建议**

- **技术栈选型**：BGE-large-zh-v1.5 + Chroma + BGE-reranker（与具体学科范围无关，为通用方案）
- **切片策略**：按章节结构切 + 递归回退
- **知识库具体内容范围**：待与大家讨论确定 MVP 用户 / 学科聚焦后确定

| 知识库范围 | 待大家讨论后确定 | MVP 用户 / 学科聚焦决策的下游问题，本调研不预先收窄 |
| 语音通道 | 后置至 V1.x | MVP 阶段先打通文字闭环 |

## 八、调研结论与产品建议

### 8.1 核心结论

1. **考研市场对"高效学习方法"的需求强烈**：传统输入式学习存在明显效率瓶颈，输出训练不足是主要原因之一
2. **费曼学习法是解决考研核心痛点的有效方法**：但传统应用形式存在"缺陪练、时间贵、无评估、无闭环"四类局限
3. **AI Agent 技术能够系统性回应上述四类局限**：特别在"陪练 + 评估 + 闭环"三个环节具备工程可行性
4. **考研市场目前缺乏专门的"AI + 费曼"垂直产品**，**存在产品机会**

### 8.2 暂时项目定位的初步建议

暂时定位为：

> **本项目以考研用户为首期落地场景，目标是构建一个基于费曼学习法的 AI 伴学引擎。**
>
> 系统在架构层面将"领域无关的核心引擎（追问、诊断、评估、复习）"与"领域相关的知识库与 Prompt 模板"做分层解耦——

### 8.3 候选功能优先级初稿

> **说明**：以下为基于本周调研的功能候选清单及优先级初稿。**最终 MVP 功能范围请共同确定。**

- **方向一（核心闭环）**：智能费曼陪练 + 自动诊断报告 + 个性化复习闭环
- **方向二（重要扩展）**：主观题费曼批改、错题费曼复盘
- **方向三（进阶能力）**：跨考零基础入门、学习进度跟踪与成长分析、同伴对练
