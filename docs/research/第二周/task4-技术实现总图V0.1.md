## 二、关键技术清单与选型理由

本期 MVP 涉及 **6 类核心技术**，每类技术对应明确的业务问题与开发难点：

| 技术 | 选型 | 解决的核心问题 |
|------|------|------------|
| **大语言模型 API** | DeepSeek-V4 Pro / Flash（备用：通义千问） | 自然语言理解、追问生成、评估打分的底层能力 |
| **Agent 编排框架** | LangGraph（基于 LangChain） | 控制费曼对话的多轮交互流程，实现"讲解→诊断→追问→评估→报告"状态机 |
| **RAG 检索增强生成** | LangChain RAG + Chroma + BGE-large-zh-v1.5 + BGE-reranker（V2） | 让 AI 评价基于教材原文，避免胡乱评价；为追问提供上下文 |
| **Prompt Engineering** | 分层 System Prompt + Few-shot 示例库 + Function Calling | 控制 AI 扮演"零基础小白"角色、定义评价标准、保证结构化输出 |
| **向量数据库** | Chroma | 存储教材切片向量，支持语义检索；为知识点推荐和漏洞溯源提供底层能力 |
| **知识图谱（V2）** | Neo4j 或同类图数据库 | 组织知识点的前驱 / 后继 / 并列关系，识别碎片化记忆、辅助漏洞定位 |

> **说明**：以上为本期 MVP 与近期 V2 的选型；语音（ASR / TTS）、双模型核验、个人记忆向量库等进阶能力延后至 V2 / V3，本图暂不展开。

**图的核心要点**（要在最终图里突出）：
1. **LangGraph 是中枢**：所有技术都围绕它工作
2. **四个外围能力**：Prompt 模板、RAG、Function Calling、LLM API —— Agent 在每个状态节点会按需调用
3. **RAG 依赖向量库**：Chroma + BGE 构成的检索底层
4. **数据持久化是终点**：MySQL + Redis + 对象存储分工

## 四、一次费曼对话的技术调用链（时序视角）

为更清晰展示技术协作时机，按一次完整对话拆解：

| 步骤 | 用户动作 | 触发的技术调用 |
|------|---------|------------|
| 1 | 选择"剩余价值"作为讲解概念 | 从 MySQL 读知识点元数据 |
| 2 | 用户文字讲解 | 文本写入 Redis 会话上下文 |
| 3 | 进入 LangGraph **诊断节点** | 拉 Prompt 模板（System + Few-shot）→ 调用 RAG 检索教材原文 → 拼接上下文 → 调 DeepSeek（Function Calling 强制结构化输出）→ 拿到诊断结果 JSON |
| 4 | LangGraph 判断有漏洞，跳到 **追问节点** | 用诊断结果作为输入 → 拉追问 Prompt 模板 → 调 DeepSeek → 输出追问问题 |
| 5 | 用户二次讲解（回到步骤 2-4 循环） | 同上，直至无漏洞或达到追问上限 |
| 6 | 进入 LangGraph **报告节点** | 拉报告模板 → 综合本轮所有诊断 / 追问 / 用户回答 → 调 DeepSeek 生成三维报告 → 写 MySQL |
| 7 | 报告页展示 | 前端拉 MySQL，渲染 Markdown + 教材原文溯源 |

**几个关键映射要点**：

- **难点 3（规避幻觉）的主力是 RAG + Function Calling**：RAG 把 AI 输出锚定在教材原文，Function Calling 让输出严格按 JSON Schema，两者结合能把大部分胡乱评价拦在外面
- **难点 5（闭环搭建）几乎全靠 LangGraph**：状态机本身就是闭环的工程化实现
- **难点 4（教材溯源）由"向量库 + RAG"双层承担**：Chroma 提供检索能力，RAG 流程把检索结果可控地拼接给 LLM
- **难点 1（真假理解判断）是综合解**：单一技术解决不了，需要 Prompt 分层校验 + RAG 上下文比对 + LLM 评估三者配合



## 六、总结

**LangGraph 状态机做大脑，DeepSeek 做能力，RAG + Chroma + BGE 做知识，Prompt + Function Calling 做约束，MySQL + Redis + 对象存储做持久化** —— 6 类技术围绕"费曼对话闭环"协作，分别对应 Task 3 的 6 大难点。∑
