# 费曼伴学智能体 MVP 产品需求文档 \(V2\.3\)

## 一、产品概述与目标

### 1\. 产品定位

基于费曼学习法的 AI 伴学工具，通过「小白听众式连续追问」倒逼学习者主动输出讲解，精准定位知识盲区，最终输出量化诊断报告，帮助学习者高效补全知识漏洞。本版本为**单知识点 MVP 验证版**，聚焦核心流程跑通与效果验证。

### 2\. MVP 核心目标

以极简工程实现跑通「AI 费曼伴学」核心闭环，剔除账号、历史存储、文档解析等非核心模块，采用单屏对话流 \+ 静态注入考点的方式，集中验证两大核心能力：

1. 大模型通过多轮追问识别用户知识漏洞的有效性

2. 结构化诊断报告的参考价值与落地可行性

### 3\. 目标用户

计算机专业备考学生、自学算法的入门学习者

### 4\. 量化成功指标

1. **流程完成率**：用户完整走完对话流程并生成报告的比例 ≥ 80%

2. **追问有效率**：AI 追问命中用户真实知识漏洞的比例 ≥ 70%（人工标注验证）

3. **系统稳定性**：接口请求成功率 ≥ 95%，异常兜底触发率 ≤ 5%

4. **格式合规率**：大模型输出 JSON 格式解析成功率 ≥ 98%

## 三、主成功流程与异常分支 \(User Flow\)

### 1\. 主成功流程 \(Main Success Scenario\)

1. **页面初始化**：用户进入页面，前端生成唯一 `session_id`，页面加载初始引导语：「请你向我讲解一下 Dijkstra 算法的核心原理，讲得越详细越好哦～」

2. **用户输入**：用户键入讲解文本，点击发送 / 按 Enter 提交

3. **请求处理**：前端禁用输入框与发送按钮，聊天区追加 AI 加载气泡，自动滚动到底部，向后端发送请求

4. **评估与决策（循环）**：

    - 后端接收请求，拼接该 session 历史对话，通过 Function Calling 调用大模型评估用户讲解

    - 若识别到逻辑漏洞，且当前追问轮次 \< 3，返回 `next_action: "follow_up"` 与追问话术

    - 前端解锁输入框，用户继续作答（重复步骤 2\-4）

5. **对话熔断**：满足「讲解完整正确」或「达 3 轮追问上限」任一条件，后端返回 `next_action: "generate_report"`，附带总结文案、卡片预览数据与完整报告 JSON

6. **结果展示**：前端渲染底部总结卡片，用户点击卡片查看完整四维诊断报告，本轮复盘结束

### 2\. 异常分支 \(Alternative Flows\)

#### 分支 1：网络超时 / 接口服务报错

- **触发条件**：接口请求超过 15 秒未响应，或后端返回 500/503 错误

- **处理机制**：前端捕获异常，解除输入框锁定，在聊天区弹出系统提示气泡：「网络似乎开小差了，请重试一下。」；本次请求不计入追问轮次

#### 分支 2：大模型返回格式解析失败

- **触发条件**：后端拿到大模型输出，但无法解析为约定 JSON 结构

- **处理机制**：后端捕获 `JSONDecodeError`，强制返回 `next_action: "follow_up"`，兜底文本：「我刚才有点走神，你能换个说法再解释一下吗？」；本次不计入追问轮次

#### 分支 3：用户输入偏离主题

- **触发条件**：用户输入内容与 Dijkstra 算法完全无关

- **处理机制**：大模型识别后返回 `next_action: "guide_topic"`，引导用户回到知识点讲解；本次不计入追问轮次

#### 分支 4：用户表示不会 / 无有效讲解内容

- **触发条件**：用户输入「不知道」「不会」等无有效信息内容，或连续 2 轮无法给出有效讲解

- **处理机制**：

    1. 首次触发：给出引导性提示线索（如「可以先试着说说这个算法是用来解决什么问题的？」），不直接给出完整答案，本次不计入追问轮次

    2. 二次触发：给出该知识点核心要点精简提要，末尾引导「现在你可以试着用自己的话再讲解一次」；本次不计入追问轮次，用户后续讲解计入正式轮次

#### 分支 5：用户重复提交 / 并发请求

- **触发条件**：请求未返回时，用户多次点击发送 / 按回车

- **处理机制**：前端防抖锁拦截，请求未返回期间输入框与按钮持续禁用，不发起二次请求

## 五、核心逻辑注入 \(System Prompt 与静态 RAG\)

以下内容为业务基准，后端需直接作为 System Prompt 写入大模型接口请求，**禁止大模型直接输出答案、禁止泄露基准事实**。

> 输出强制采用 Function Calling 机制，大模型必须调用约定的结构化输出函数，后端对函数返回值做二次格式校验，确保 100% 符合 JSON Schema
> 

```Plain Text
你现在是一个零基础、但充满好奇心的小白听众。你的任务是听用户讲解「数据结构 - Dijkstra 算法」，通过逻辑推演找出他表述中的漏洞，用提问引导用户自己发现错误。

【后台判分基准事实（Ground Truth，绝对禁止直接告诉用户）】
1. 概念前提：图的所有边权必须为非负数；负权会破坏「已标记节点最短路径已确定」的核心结论
2. 核心机制：贪心策略。每次从未访问节点中选出距离起点最近的节点，标记为已访问，并用该节点的距离松弛（更新）其所有相邻节点的距离
3. 原理证明：依赖三角不等式与非负权前提——因为边权非负，后续不可能通过其他未访问节点得到更短的路径，因此当前距离最短的节点，其最短路径已确定
4. 常见误区：无法解释贪心策略的正确性、忽略非负权前提、混淆「松弛操作」的作用、认为可以处理负权图

【对话与轮次规则】
1. 最多发起3轮追问，第3轮追问后的下一次用户输入，无论是否完整，都必须生成最终报告
2. 若用户讲解已完整覆盖所有基准事实且无逻辑错误，可提前结束，直接生成最终报告
3. 若用户输入与Dijkstra算法无关，请引导用户回到主题，本次不计入追问轮次
4. 若用户表示不会、不知道，先给出引导性线索；若仍无法作答，再给出精简核心要点并引导重新讲解，本次不计入追问轮次

【行为规则 - 严格执行】
1. 绝对禁止直接给出完整正确定义、答案或原理解释，禁止泄露基准事实的原文
2. 你只能通过「提问」「举反例」「给出线索提示」引导用户自己发现错误
3. 每次回复只挑出当前最严重的1个逻辑漏洞进行单点追问，不要一次性提多个问题
4. 语气保持好奇、友好，符合小白听众人设，不要说教

【输出要求 - 必须严格遵守】
1. 必须通过 Function Calling 输出结构化结果，禁止自由文本输出
2. 严格遵循下方 JSON Schema 结构，字段名、枚举值、分值范围完全一致
3. 生成报告时，得分必须基于用户的真实讲解情况，对应四个评估维度的实际表现

【JSON Schema】
{
  "next_action": "follow_up / generate_report / guide_topic 三选一",

  "card_preview": {
```

## 六、前后端数据契约 \(API Schema\)

**接口路径**: `POST /api/v1/feynman/chat`
**请求 Header**: `Content-Type: application/json`
**超时阈值**：前端设置 15s 超时
**输出管控**：后端通过 Function Calling 强制大模型输出结构化 JSON，降低格式异常概率

### 1\. Request Body \(前端 \-\> 后端\)

```json
{
  "session_id": "uuid字符串，页面加载时前端生成，全流程唯一",
  "user_input": "用户输入的讲解文本，字符串类型"
}
```

### 2\. Response Body \(后端 \-\> 前端\)

所有响应均为统一结构，后端需强制保证 JSON 格式合规：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "next_action": "follow_up",
    "reply_text": "听懂了更新距离！但如果图里面有的边距离是负数，这个方法还管用吗？",
    "card_preview": null,
    "final_report": null
  }
}
```

#### 字段说明

|字段名|类型|枚举值 / 说明|
|---|---|---|
|`next_action`|string|`follow_up`：继续追问；`generate_report`：生成最终报告；`guide_topic`：引导回主题|
|`reply_text`|string|AI 直接展示给用户的话术内容|
|`card_preview`|object|报告卡片预览数据，`next_action=generate_report` 时返回，否则为 `null`，总分范围 0\-40|
|`final_report`|object|完整四维诊断报告，`next_action=generate_report` 时返回，否则为 `null`|

### 3\. 错误码定义

|code|说明|前端处理逻辑|
|---|---|---|
|200|请求成功|按正常流程渲染|
|400|参数错误（如空输入）|解除输入锁定，弹出对应提示|
|500|服务端内部错误|解除输入锁定，提示网络异常请重试|
|503|大模型服务不可用|解除输入锁定，提示服务繁忙请稍后|

### 验收标准

1. **流程闭环**：从进入页面到生成完整报告，全流程无阻塞、无报错

2. **规则合规**：AI 全程不直接给出完整答案，追问为单点提问，最多 3 轮追问后强制熔断

3. **异常可用**：模拟超时、格式错误、无关输入、用户不会四类异常，系统均能正常兜底不崩溃

4. **数据合规**：所有接口响应严格遵循契约，字段名、类型、枚举值、分值范围无偏差

## 七、非功能需求

1. **性能**：单接口平均响应时间 ≤ 10s（含大模型调用），最大超时阈值 15s

2. **兼容性**：MVP 仅适配桌面端 Chrome、Edge 最新版浏览器，不强制移动端适配

3. **安全**：用户输入做基础敏感词过滤；无账号体系，所有会话数据仅内存临时存储，页面刷新后数据清空

4. **可用性**：所有异常场景均有用户可理解的文案提示，无空白、卡死状态

## 八、风险与应对方案

|风险点|影响等级|应对方案|
|---|---|---|
|大模型输出格式不稳定，解析失败|中高|1\. Function Calling 强制结构化输出<br>2\. 后端增加 JSON 正则提取 \+ 格式校验兜底<br>3\. 解析失败触发兜底话术，不计入轮次|
|AI 追问质量差，偏离知识点|中|1\. 静态基准事实强约束评判范围<br>2\. 上线前人工验收 10 组测试用例，不达标则迭代 Prompt|
|用户恶意输入无关内容|低|1\. 增加主题偏离检测，引导回正题<br>2\. 连续 3 次偏离则终止对话，提示请围绕知识点讲解|
|用户完全不会知识点无法推进|低|1\. 阶梯式引导：先给线索，再给精简要点<br>2\. 引导重新讲解后计入正式轮次，保证流程可闭环|

### 存储说明

```json
{

  "category": "数据结构-图论",
  "ground_truth": {
    "concept_prerequisite": {
      "name": "概念前提",
      "content": "Dijkstra算法适用于边权非负的带权图，所有边的权重必须大于等于0。负权边会破坏算法的核心结论——已标记为已访问的节点，其最短路径已经确定，因此算法无法正确处理存在负权边的图。"
    },
    "core_mechanism": {
      "name": "核心机制",
      "content": "算法基于贪心策略实现，执行步骤为：1. 初始化起点距离为0，其余节点距离为无穷大；2. 每次从未访问的节点中，选择当前距离起点最近的节点；3. 将该节点标记为已访问，并用它的当前距离去松弛（更新）其所有相邻节点的距离；4. 重复上述步骤，直到所有节点都被访问或目标节点被访问。"
    },
```
